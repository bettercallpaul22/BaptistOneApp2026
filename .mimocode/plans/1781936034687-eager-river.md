# Fix: Access Switch Not Called After Login

## Root Cause

The access switch (`switchAccessThunk`) is dispatched inside a `useEffect` in `usePostLoginAccess`, but the LoginPage's navigation effect runs in the **same render commit** and reads the **stale** `switchState` (still `'idle'`), making `ready = true` and navigating to `/home` before the switch completes.

Sequence:
1. `loginThunk` succeeds → `authData` set → re-render
2. Effect A (`usePostLoginAccess`): calls `switchDispatch({ type: 'start' })` — queues state update
3. Effect B (`LoginPage`): reads `ready` from current render → `switchState.status` is still `'idle'` → `ready = true` → navigates to `/home`
4. React processes batched update → `switchState.status = 'switching'` → but component already navigated away

## Fix: Dispatch switch in `onSubmit` before navigation

Move the access switch to happen synchronously inside the login flow (`onSubmit` handler), so it completes before any navigation effect fires.

### Files to modify:

**1. `src/pages/auth/login/LoginPage.tsx`**
- After `loginThunk` succeeds, check if `currentAccess.resourceType !== 'church-member'` and user has a `church-member` access
- If so, dispatch `switchAccessThunk` with the church-member access ID
- This happens inside `onSubmit` (which is `async` and `await`s), so it completes before the navigation `useEffect` reads `ready`

**2. `src/pages/auth/email-verify/EmailVerifyPage.tsx`**
- Same pattern: after `handoffLoginThunk` succeeds, check and switch if needed

**3. `src/hooks/usePostLoginAccess.ts`**
- Simplify: remove the effect-based switch logic entirely
- Keep only the `ready` / `showNoAccessModal` / `registerRequired` state tracking for edge cases
- Since the switch now happens in the login handlers, this hook mainly serves to detect `register-required` state and provide logout functionality
- `ready` should be `true` as long as `authData` is present and access is valid (switch already happened in onSubmit)

### Detailed changes:

**LoginPage.tsx `onSubmit`:**
```ts
const onSubmit = useCallback(
  async ({ email, password }: LoginForm) => {
    try {
      const authResult = await dispatch(loginThunk({ email, password })).unwrap();

      // Switch to church-member access if available and not already active
      if (authResult.currentAccess.resourceType !== 'church-member') {
        const churchMemberAccess = authResult.userAccess.find(
          (a) => a.status === 'active' && a.resourceType === 'church-member',
        );
        if (churchMemberAccess) {
          await dispatch(switchAccessThunk({ accessId: churchMemberAccess.id })).unwrap();
        }
      }

      dispatch(pushNotification({ type: 'success', title: 'Signed in', message: 'Welcome back to BaptistOne.' }));
    } catch (error) {
      const message = (error as { message?: string })?.message ?? 'Check your email and password, then try again.';
      dispatch(pushNotification({ type: 'error', title: 'Sign in failed', message }));
    }
  },
  [dispatch],
);
```

**EmailVerifyPage.tsx** — same pattern after `handoffLoginThunk`.

**usePostLoginAccess.ts** — Simplify to:
- `ready` = true when `authData` is present and `currentAccess` is valid (church-member OR profile with church-member available, since switch already happened)
- Remove the `useEffect` that dispatched the switch
- Keep `isRegisterRequired` detection and `handleLogoutAndRedirect`

## Verification
1. Login with a user whose `currentAccess` is `profile` but has a `church-member` access → should switch to church-member, then navigate to home
2. Login with a user whose `currentAccess` is already `church-member` → should go straight to home
3. Login page with no auth → should NOT fire any switch request (401 error should be gone)
4. Email verify flow → same behavior as login
